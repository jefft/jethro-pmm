# Permissions

Jethro controls what each staff member can do through a set of permissions. Permissions are assigned per user account and can be changed at **Admin → User Accounts**.

## Permission hierarchy

Higher permissions automatically include lower ones within the same category. For example, granting `EDITROSTER` also grants `VIEWROSTER`. The hierarchy is:

<table>
  <thead><tr><th>Category</th><th>Hierarchy</th></tr></thead>
  <tbody>
    {(() => {
      // Group hierarchy edges by the category of the "to" permission
      const catEdges = {};
      for (const edge of data.hierarchy) {
        const cat = data.permissions[edge.to].category;
        if (!catEdges[cat]) catEdges[cat] = [];
        catEdges[cat].push(edge);
      }
      // Build chains for each category
      const rows = [];
      for (const cat of data.categoryOrder) {
        const edges = catEdges[cat];
        if (!edges || edges.length === 0) continue;
        // Build chain: find the leaf-most permission, walk backwards
        const children = new Set(edges.map(e => e.to));
        const parents = new Set(edges.map(e => e.from));
        const leaves = [...children].filter(c => !parents.has(c));
        const chains = leaves.map(leaf => {
          const chain = [leaf];
          let current = leaf;
          while (true) {
            const edge = edges.find(e => e.to === current);
            if (!edge) break;
            chain.unshift(edge.from);
            current = edge.from;
          }
          return chain;
        });
        // If one chain covers all, show it spanning categories
        if (cat === 'System Administration') {
          rows.push(<tr key={cat}><td>{cat}</td><td><code>SYSADMIN</code> (includes all permissions)</td></tr>);
          continue;
        }
        const chainStrs = chains.map(c => c.map(k => <code key={k}>{k}</code>).reduce((a, b) => <>{a} → {b}</>));
        rows.push(<tr key={cat}><td>{cat}</td><td>{chainStrs.map((c, i) => <div key={i}>{c}</div>)}</td></tr>);
      }
      return rows;
    })()}
  </tbody>
</table>

Permissions in different categories are independent — having `MANAGEREPORTS` does not give you `EDITPERSON`, and so on. SMS permissions (`SENDSMS`, `VIEWSMS`), People (`EDITPERSON`), and Sermons (`EDITREC`) are standalone and do not form hierarchies.

New staff accounts receive the following permissions by default:

  <li><strong>Included:</strong> {Object.entries(data.permissions).filter(([, p]) => p.default).map(([k]) => <code key={k}>{k}</code>).reduce((a, b) => <>{a}, {b}</>)}</li>

The default can be changed at **Admin → System Configuration** → `DEFAULT_PERMISSIONS`. Changes only affect new accounts — existing users are not modified.

## Feature-flag gating

Some permissions require a feature to be enabled in `ENABLED_FEATURES`. If the feature is turned off, the associated permissions have no effect — even if granted on a user account:

<table>
  <thead><tr><th>Feature</th><th>Affected permissions</th></tr></thead>
  <tbody>
    {Object.entries(data.features).map(([featureName, feature]) => (
      <tr key={featureName}>
        <td>{featureName}{feature.note ? <><br /><small>{feature.note}</small></> : null}</td>
        <td>{feature.permissions.map(k => <code key={k}>{k}</code>).reduce((a, b) => <>{a}, {b}</>)}</td>
      </tr>
    ))}
  </tbody>
</table>

Permissions not listed above (People, SMS, Reports, Groups, System Administration) are always active regardless of feature flags.

## Congregation and group restrictions

Separate from permission levels, user accounts can have **congregation restrictions** and **group restrictions**. These limit which people, families, and groups a user can see, regardless of their permission level. SysAdmins bypass these restrictions.
